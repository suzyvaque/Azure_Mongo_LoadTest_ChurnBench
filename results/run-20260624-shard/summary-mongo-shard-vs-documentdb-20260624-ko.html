샤딩 MongoDB vs DocumentDB — Connection Churn 벤치마크 요약

테스트 run-20260624-shard · 양쪽 모두 TLS 적용 · 각 DB와 동일 AZ VM에서 단일 실행 · 각 시나리오 3회 반복 × 600초 · 값은 3회 반복 백분위수의 평균(p50 / p90 / p99), 단위 ms.

1. 테스트 시나리오

  • 대상 워크로드: HPC Connection Churn으로 매 작업(Task)마다 새 연결을 열고 작업 후 끊는 단일 사용 연결 패턴(연결 풀링/재사용 없음)
  • 부하 시나리오:
    • Steady: 일정한 135 Tasks/s
    • Burst: 푸아송 도착(λ=0.57), Job당 14~500 Tasks의 스파이크
  • 워크로드:
    • 단일 opfind-input: Input Collection에서 읽기 수행
    • 단일 opinsert-output: Output Collection에 쓰기 수행
    • 전체 4opfind→remove→insert→find: Input Collection에서 읽기 → Output Collection에서 삭제 → Output Collection에 쓰기 → Output Collection에서 읽기. 각 작업 사이에 10초 pseudo-sleep 포함.
  • 비교 대상: 자체 운영 샤딩 MongoDB vs 관리형 Azure DocumentDB
  • 클라이언트 모델: 작업당 새 MongoClient 1개(maxPoolSize=1), 2개 mongos에 라운드로빈 단일 서버 직결(directConnection=true)

2. 리소스 사양

  • MongoDB: Shard 클러스터 2개, mongos 라우터 2개, config server 1개. hashed ReqId로 샤딩(샤드 간 ~50/50 분산), 서버측 TLS(allowTLS, 자체 서명 인증서, 클라이언트 tlsInsecure=true)
  • Azure VM: Windows Server, E32as v6 (32 vcpus, 256 GiB memory)
  • DocumentDB: Azure DocumentDB(vCore), HA 활성화, 관리형 TLS.
  • 공통 스택: MongoDB Server 7.0 / wire 7.0, .NET 8(LTS), MongoDB C# Driver 2.30
  • 부하 생성기: Windows Server, D8ds v5 (8 vcpus, 32 GiB memory)

3. 핵심 지표: 연결(Connection) vs 작업(Operation)

  • 각 작업은 새 연결을 사용하며, 드라이버는 지연(lazy) 연결이라 첫 DB 작업 시 TCP+TLS+인증이 함께 발생
  • Connection(TCP+TLS+auth) = ConnectionOpenMs: 핸드셰이크 시간으로, 연결 비용은 여기에만 집계
  • Operation 시간: 연결 시간을 제외한 서버 실행 시간
    • cold op = 새 연결의 첫 작업으로, op − ConnectionOpenMs 시간으로 표시
    • warm op = 이미 열린 소켓의 작업으로, 순수 서버 실행 시간

5. 결과 요약

  • MongoDB의 샤딩이 요구됨: 기존 단일 mongo-vm은 전체 워크로드에서 22.8%(steady)/32.0%(burst)의 작업이 실패
  • 연결 비용을 제외하면 작업 자체는 두 DB 모두에서 빠름: 단일 operation은 MongoDB ~19–22ms p50, DocumentDB ~12–13ms p50. 쿼리가 느리다는 인상은 cold 연결 핸드셰이크가 측정 시간에 포함됐기 때문
  • [DocumentDB 강점] Steady 중앙값 우위: 모든 steady 행에서 연결·작업 p50/p90이 약 2배 빠름(핸드셰이크 ~12ms vs ~22ms).
  • [DocumentDB 강점] warm 작업 우위: 소켓이 열린 뒤 remove/insert/find가 2–10배 빠름(sub-6ms p50 vs 4–29ms), burst에서도 tail이 더 작음
  • [DocumentDB 강점] burst 처리량·오류율: burst 전체 워크로드에서 처리량 135.3 vs 128.4 req/s, 오류 0.033% vs 0.16%로 우위
  • [MongoDB 강점] burst 연결 tail: 2×mongos 팬아웃으로 burst 연결 p90/p99가 더 좋음(전체 워크로드 p90 1,059 vs 1,606ms, p99 2,022 vs 3,236ms)
  • HPC는 연결 재사용이 없는 Connection Churn 패턴이므로 연결 frontend(핸드셰이크 처리량)가 중요한 요소

6. 부록: 비용 분석

성능은 유사(DocumentDB가 약간 우위)하지만, HA를 반영한 운영 비용에서는 관리형 DocumentDB가 유리. 실제 운영 시에는 MongoDB HA를 위해 인스턴스 수가 2배로 늘어나며, 패치·백업·모니터링 등 운영 부담 및 일부 지연 시간도 추가됨.
