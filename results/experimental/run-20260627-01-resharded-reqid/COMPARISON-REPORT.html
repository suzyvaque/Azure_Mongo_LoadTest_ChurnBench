2. 핵심 결론 (Key findings)

  • 컬렉션을 {ReqId:"hashed"} 로 sharding 해도 latency 와 throughput 에 미치는 영향은 사실상 없었습니다. 두 DocumentDB 구성의 차이는 run 간 변동(run-to-run variance) 범위 안에 있습니다.
  • 이유: Azure DocumentDB (Cosmos vCore) 는 shardCollection 시점에 logical shard key 만 정의하며, preview 단계의 rebalancer (by_disk_size 전략)는 작고 균일하게 저장된 데이터셋을 물리적으로 재분배하지 않습니다. resharding + reseed 이후에도 explain() 결과 100k 문서가 단일 physical shard 에 그대로 남아 있음을 확인했습니다.
  • resharding 후 ReqId 기준 find_input 은 router 가 ReqId 를 hashing 하는 targeted single-shard 쿼리가 되지만, 모든 데이터가 한 shard 에 있으므로 routing 이점은 관측되지 않습니다.
  • connection 처리(churn rate, peak concurrency)는 shard key 와 무관 합니다. 이는 데이터 배치가 아니라 client host 와 cluster gateway 에 의해 결정됩니다.

5. 주의사항 (Caveats)

  • resharded run 은 실제로 데이터가 분산된 2-shard topology 를 나타내지 않습니다. 데이터는 물리적으로 한 shard 에 남아 있습니다 (Cosmos vCore preview rebalancer 의 제약). 물리적 분산을 강제하려면 (1) by_disk_size 를 trigger 할 만큼 큰 storage 불균형, (2) 노출되지 않은 by_shard_count 전략, 또는 (3) 생성 시점부터 shardCount=2 인 새 cluster 로 seeding 하는 방법이 필요합니다.
  • Burst 의 find_input p99 차이는 데이터 배치가 아니라 client 측 saturation (CPU 약 45%, peak 약 13.6k 동시 socket) 에 의해 주도됩니다.
  • 모든 run 은 동일한 client host, network path (private endpoint), M80 cluster tier 를 공유합니다.
