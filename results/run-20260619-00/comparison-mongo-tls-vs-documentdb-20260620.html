MongoDB Connection-Churn Benchmark — mongo-vm (TLS) vs DocumentDB

Campaign run-20260619-00 · TLS-vs-TLS · 3 iterations × 600 s per run · steady (135 Tasks/s) and burst (Poisson λ=0.57) run as separate runs · latency in ms, mean of 3 iterations' percentiles.

Workload: new MongoClient per Task (no reuse; maxPoolSize=1/minPoolSize=0). Three workloads: single-op find-input, single-op insert-output, full 4-op full-workload (find→remove→insert→find).
TLS: both backends do a full handshake per connection — mongo-vm server-side TLS enabled (allowTLS, self-signed cert + CAFile); documentdb always-TLS. Stack: MongoDB 7.0 / wire 7.0 · .NET 8 · C# Driver 2.30.

How the metrics are separated

Each Task opens a brand-new connection; the driver connects lazily, so the first op triggers TCP+TLS+auth. Two independent measurements are recorded:

  • Connection (TCP+TLS+auth) — ConnectionOpenMs from the driver's ConnectionOpenedEvent duration. This is the only place the handshake cost is counted.
  • Server query / execution — the operation latency net of the connection handshake. For single-op tests the one (cold) op bundles connect+query, so the server-execution figure is op − ConnectionOpenMs at the same percentile (query time with TCP+TLS+auth removed). For the full workload, op1 (find_input) is cold (shown net of connection); ops 2–4 run on the warm socket and are already pure server execution.
Approximation: percentiles are not additive, so op − connection at a given percentile is an indicative decomposition, not an exact per-request subtraction. The raw ConnectionOpenMs / OperationMs percentiles in each aggregate.json are exact. The earlier non-TLS mongo runs under mongo_tls_false/ are excluded (no TLS crypto).

Key Findings

  • Server query time is small once TCP+TLS+auth is removed. Steady single-op server query is ~18–20 ms p50 on both (find 19.7 ms mongo / 17.7 ms DocumentDB; insert 22.5 / 18.2). The "slow query" impression was the cold-connection handshake counted inside the op, not server execution.
  • With TLS equalized, steady single-op is near-parity. mongo-vm holds the tighter tail (connection & query p99 lower), DocumentDB the lower median; both 0% errors at full 135 req/s.
  • Warm full-workload ops confirm it. Warm-socket find_output is 26 ms p50 (mongo) / 1 ms (DocumentDB); only the cold op1 carries connection cost.
  • Under burst, DocumentDB is ~2× faster across connection, server-query and cycle, and stays at 0% errors where mongo-vm sheds 0.37–0.56%.
  • Full workload breaks the single VM. Client saturation (CPU 100%, 50k+ handles) drives mongo-vm to 22.8% (steady) / 32.0% (burst) failures — mostly ServerSelectionTimeout — vs DocumentDB ≤0.05%. DocumentDB wins every full-workload row.

Migration decision guide (TLS-on-both)

For light single-op churn, mongo-vm and DocumentDB are effectively tied once connection and query are measured separately — mongo-vm even wins the tail. For the full multi-op workload under churn, Azure DocumentDB (vCore) is the clear recommendation: it sustains full throughput at near-zero errors while the single mongo-vm node saturates the client and sheds 23–32% of work. The decisive factor is not server query speed (~sub-30 ms once connection is excluded) but how each backend's connection front-end copes with TLS-handshake churn at scale.
